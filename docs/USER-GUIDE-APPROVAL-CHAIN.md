# คู่มือการใช้งานกฎการอนุมัติวันลา

เอกสารนี้สำหรับ Admin โรงพยาบาลที่ต้องตั้งค่ากฎการอนุมัติวันลาใน Hospital Operations Portal (HOP)

## กฎการอนุมัติวันลาคืออะไร

กฎการอนุมัติวันลา คือ rule ที่ผูกกับผู้ใช้งานแต่ละคน และระบุลำดับผู้อนุมัติที่ระบบใช้หลังจากเจ้าหน้าที่ส่งคำขอลา เช่น หัวหน้าหน่วยงาน → ผู้อำนวยการ

ระบบยังใช้ตารางเดิม `approval_chains` / `approval_chain_steps` แต่ใน UI เรียกว่า `กฎการอนุมัติวันลา`

## ใช้เมื่อไร

ใช้เมื่อหน่วยงานต้องกำหนดว่าคำขอลาของเจ้าหน้าที่ต้องผ่านใครบ้างก่อนอนุมัติสำเร็จ

## ตัวอย่าง Flow

```text
เจ้าหน้าที่ส่งคำขอลา
↓
หัวหน้าหน่วยงานอนุมัติ
↓
ผู้อำนวยการอนุมัติ
↓
อนุมัติสำเร็จ
```

## ใครควรเป็นผู้อนุมัติแต่ละขั้น

- ขั้นที่ 1: หัวหน้าหน่วยงานหรือหัวหน้างานที่ดูแลเจ้าหน้าที่โดยตรง
- ขั้นสุดท้าย: ผู้อำนวยการหรือผู้บริหารที่มีอำนาจอนุมัติขั้นสุดท้าย
- กรณีมีหลายชั้นการบังคับบัญชา สามารถเพิ่มขั้นกลางได้ตามนโยบายของโรงพยาบาล

## วิธีสร้างกฎการอนุมัติ

1. เข้าสู่ระบบด้วยบัญชีที่มีสิทธิ์จัดการกฎการอนุมัติ
2. ไปที่เมนู `ระบบลา` → `กฎการอนุมัติวันลา`
3. กด `เพิ่มกฎการอนุมัติ`
4. ระบุชื่อ rule เช่น `IT-STAFF`
5. ระบุรายละเอียดให้อ่านง่าย เช่น `เจ้าหน้าที่ IT → หัวหน้าหน่วยงาน → ผู้อำนวยการ`
6. เลือกหน่วยงาน/ประเภทลาเป็น metadata ได้ แต่ระบบ submit จะยึด rule ที่ผูกกับ user เป็นหลัก
7. กดบันทึก

## วิธีผูก Rule กับ User

1. ไปที่เมนู `จัดการระบบผู้ใช้` → `จัดการผู้ใช้`
2. เปิดหน้าเพิ่มหรือแก้ไขผู้ใช้งาน
3. เลือกช่อง `กฎการอนุมัติวันลา`
4. กดบันทึก
5. กดปุ่ม `ทดสอบกฎการอนุมัติ` ในรายการผู้ใช้เพื่อตรวจผลลัพธ์

## วิธีเพิ่มขั้นอนุมัติ

1. เปิดรายละเอียดสายอนุมัติ
2. เพิ่มขั้นอนุมัติทีละขั้นตามลำดับ
3. ระบุชื่อขั้น เช่น `หัวหน้าหน่วยงาน`
4. เลือกผู้อนุมัติ หรือเลือก role ที่ระบบรองรับ
5. ตรวจว่า permission ของผู้อนุมัติรองรับ `LeaveApproval.ApproveCurrentStep`

## การเลือกหน่วยงานและประเภทลา

- ใน model ใหม่ หน่วยงานและประเภทลาเป็นข้อมูลประกอบของ rule
- ตอน submit ระบบไม่ให้ผู้ขอลาเลือก rule เอง
- ระบบจะอ่าน `กฎการอนุมัติวันลา` จาก profile ของ user เท่านั้น

## ตั้งให้หัวหน้าหน่วยงานเป็นขั้นที่ 1

ตั้งชื่อขั้นเป็น `หัวหน้าหน่วยงาน` และเลือกผู้ใช้งานที่เป็นหัวหน้าหน่วยงาน เช่น `head01`

## ตั้งให้ผู้อำนวยการเป็นขั้นสุดท้าย

เพิ่มขั้นสุดท้ายชื่อ `ผู้อำนวยการ` และเลือกผู้ใช้งาน role Director เช่น `director01`

## กรณีผู้อำนวยการลาเอง

ห้ามให้ผู้อำนวยการอนุมัติคำขอของตัวเอง ระบบควรมี fallback approver เช่น รองผู้อำนวยการหรือผู้รักษาการแทน และต้องบันทึก audit log เมื่อใช้ fallback

## วิธีทดสอบ

1. Login เป็นเจ้าหน้าที่ เช่น `staff01`
2. สร้างคำขอลาและส่งคำขอ
3. Login เป็น `head01` ตรวจว่ามีงานรออนุมัติ
4. อนุมัติขั้นที่ 1
5. Login เป็น `director01` ตรวจว่างานย้ายมารออนุมัติขั้นสุดท้าย
6. อนุมัติขั้นสุดท้าย
7. ตรวจสถานะคำขอเป็น `อนุมัติแล้ว`

## ข้อควรระวัง

- ห้ามให้ผู้ขอลาเป็นผู้อนุมัติคำขอของตัวเอง
- อย่าใช้ SuperAdmin อนุมัติแทนแบบเงียบ ๆ ให้ใช้ override flow เฉพาะกรณีจำเป็นและต้องระบุเหตุผล
- ตรวจ permission ของผู้อนุมัติทุกคนก่อนใช้งานจริง
- ทดสอบด้วยบัญชีจำลองก่อนเปิดใช้งานกับหน่วยงานจริง

## ตัวอย่างกฎการอนุมัติแผนก IT

```text
IT-STAFF
  staff01 / staff02
  ↓
  head01
  ↓
  director01

IT-HEAD
  head01
  ↓
  director01

DIRECTOR
  director01
  ↓
  admin_support / ผู้อนุมัติสำรอง
```
